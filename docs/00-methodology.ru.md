# Методология tms-pipeline

> Подробный разбор. Краткий обзор — в [README](../README.ru.md); английская версия —
> [00-methodology.md](00-methodology.md).

Это **методология поставки** (delivery methodology) для ИИ-агентов, пишущих код (Claude Code, Codex и
совместимые инструменты). Она задаёт, как уже сформулированная задача проходит путь от тикета до
проверенного, готового к слиянию кода. Это **не** генератор проектов, и она не придумывает за вас функции
— см. раздел «What this is / is NOT» в README.

---

## 1. Главный принцип: инженерия контекста

Универсальные промпты («сделай эту функцию, без багов») не работают. По мере роста объёма работы один
мега-промпт накапливает шум, а вывод модели деградирует, превращаясь в усложнение, баги и дыры в
безопасности.

Решение — **строгая декомпозиция**: разбить работу на стадии и сделать так, чтобы результат каждой стадии
становился **узким, очищенным от шума контекстом** для следующей. На каждом шаге агент получает ровно ту
информацию, которая ему нужна, и ограничен стандартами вашего проекта — ничего лишнего. Качество достигается
не магией модели, а дисциплинированным контролем контекста.

У этой идеи есть хорошо известная форма из четырёх фаз — **Research → Design → Planning → Implementation**
— с многоагентной реализацией в стиле «mob» и контрольными воротами качества. tms-pipeline берёт эту основу
и усиливает её до восьми стадий, добавляет аудит дизайна с оценкой серьёзности, модель стоимости запуска
агентов и явные правила, чтобы найденная по ходу работа никогда не терялась.

## 2. Восемь стадий

Каждая стадия производит ровно один долговечный артефакт в папке задачи (`docs/<TICKET-ID>/` или там, где
вы зададите `TASK_FOLDER_PATTERN`). Стадии выполняются по одной; по умолчанию агент останавливается после
каждой и ждёт вашего подтверждения (вы можете попросить его пройти весь путь целиком).

| # | Стадия | Навык | Результат | Что происходит |
|---|-------|-------|--------|--------------|
| 00 | Тикет | `/tms-ticket` | `00_ticket.md` | Зафиксировать драйвер, объём, критерии приёмки; подтвердить, что задача есть в бэклоге; классифицировать режим задачи. |
| 01 | Исследование | `/tms-research` | `01_research.md` | Сузить кодовую базу до фактов, которые имеют значение («как есть»), через ограниченное параллельное расхождение поиска (search fan-out). Без мнений, без советов по рефакторингу. |
| 02 | Дизайн | `/tms-design` | `02_design.md` | Составить единый контракт дизайна — минимально достаточное изменение на владеющем слое. Проверяется человеком до написания любого кода. |
| 02b | Аудит пробелов | `/tms-gap-audit` | `02b_gap_audit.md` | Один ограниченный состязательный проход по дизайну, с иной аналитической оптики, классифицирующий пробелы по серьёзности. |
| 03 | План поставки | `/tms-plan` | `03_delivery_plan.md` | Разбить дизайн на небольшие, независимо выпускаемые волны; пометить каждую профилем сопровождения. |
| 04 | Реализация | `/tms-implement` | `04_implementation.md` | Многоагентный «mob»: ведущий оркеструет рабочих + проверяющих агентов волна за волной, с контрольными воротами качества. |
| 05 | Отчёт о тестировании | `/tms-test` | `05_test_report.md` | Проверить первичный (видимый пользователю) сигнал плюс вторичные сигналы (тесты, типы, линтер, сборка). |
| 06 | Контрольные ворота проверки | `/tms-review` | `06_review_gate.md` | Сверить реализацию с контрактом дизайна; вынести вердикт go / conditional_go / no-go. |

Нет ни стадии мозгового штурма, ни стадии генерации идей. Решать, *что* строить, — ваша задача, и делаете вы
это по-своему; конвейер запускается, как только задача существует.

### Поэтапное выполнение
Если вы запускаете задачу по ID тикета, агент завершает только запрошенную стадию и останавливается для
подтверждения. Он не перескочит из исследования/дизайна сразу в код, как и из реализации в
тестирование/проверку, без вашего разрешения.

### Интервью на стадии исследования
После исследования, если работа выявляет реальные продуктовые/операционные выборы или выборы по объёму, агент
задаёт вам интервью **в чате** перед дизайном — сформулированное под вашу аудиторию, с вариантами на простом
языке и рекомендацией. Неотвеченные вопросы никогда молча не записываются в дизайн как решения.

## 3. Три вещи, которых нет у большинства конвейеров агентов

Эти части стоит перенять, даже если вы сохраните собственные названия стадий.

### 3.1 Аудит пробелов с оценкой серьёзности (стадия 02b)

Большинство конвейеров идут дизайн → код → проверка. tms-pipeline вставляет **ограниченный состязательный
аудит самого дизайна, до написания любого кода**, выполняемый с иной оптики, чем использовал автор дизайна
(чередуйте: безопасность / параллелизм / UX / эксплуатация / целостность данных / приватность). Каждый
пробел классифицируется ровно в одну категорию серьёзности — с явными правилами против завышения:

- **A — Блокер:** потеря данных, нарушение безопасности, нарушение приватности/комплаенса, дублирование
  боевых данных / нарушение целостности или блокировка запуска. Исправляется прямо в дизайне до планирования.
- **B — Инцидент:** восстановимый боевой инцидент (зависшая задача, пропущенное уведомление, неполный
  откат). Исправляется в дизайне или передаётся в планирование с пометкой об обработке.
- **C — Полировка:** шероховатости UX, неполная локализация (i18n), отсутствие метрик/runbook. Фиксируется
  как *сгруппированные* тикеты в бэклоге.
- **D — Теоретическое:** низкая вероятность / радиус поражения. Заносится в бэклог только если исправление
  очевидно и дёшево; иначе отбрасывается с однострочным обоснованием.

И **критерии остановки**, чтобы аудит никогда не превращался в переработку дизайна: не более двух проходов
(второй — только если первый нашёл класс A); чистый проход без A и без B останавливает аудит;
преимущественно C/D — останавливает. Класс A означает «ломает запуск или утекают данные», а не «можно было
бы сделать получше» — рубрика существует, чтобы *предотвращать* завышение.

### 3.2 Модель стоимости агентов: профили сопровождения

Запускать полную команду проверяющих агентов на каждое изменение дорого. tms-pipeline классифицирует каждую
волну реализации в один из трёх **профилей сопровождения** и по умолчанию выбирает самый дешёвый, который
допускают правила:

- **Профиль A — Минимальный** (Developer + Tester + Reviewer): переименования, текст/i18n, стилизация,
  волны только с тестами.
- **Профиль B — Стандартный** (+ Architect): нетривиальная бизнес-логика, изменения формы API, изменения
  схемы, не затрагивающие аутентификацию/мультиарендность.
- **Профиль C — Полный** (+ Security): всё, что касается аутентификации, разграничения по
  арендатору/идентичности, валидации ввода на границе доверия, секретов/подписи, платежей, PII или новых
  изменяющих данные командных поверхностей.

Профиль C обязателен для своих триггеров (точные триггеры вашего проекта вы перечисляете в `AGENTS.md`), но
это *исключение*, а не значение по умолчанию. Минимальное сопровождение стоит примерно 40% от полного на
волну — выбор профиля является главным рычагом стоимости, а «запустить полное сопровождение на всякий случай»
явно не одобряется.

### 3.3 Ничто найденное не теряется: фиксация последующих задач и шагов запуска

Работа, найденная посреди задачи, обычно испаряется в чате. tms-pipeline делает фиксацию жёстким правилом с
таблицей маршрутизации:

- **Действенные последующие задачи** → новая строка в бэклоге.
- **Расхождение документации / пробелы спецификации** → обновить владеющий документ-источник истины +
  перекрёстная ссылка в бэклоге.
- **Ручные предзапусковые действия** (применить миграцию, задать ключ окружения, прогнать живой smoke,
  ограничение на порядок деплоя, любое условие `conditional_go`) → **playbook запуска**, написанный так,
  чтобы не-инженер мог выполнить их без посторонней помощи (точные шаги, копируемая команда, точный критерий
  прохождения, предусловие).
- **Поздние архитектурные решения** → ADR/заметка с обратной ссылкой из бэклога.

И дисциплина, сохраняющая бэклог пригодным к использованию: **группируйте, не дробите.** Бэклог, разбитый на
десятки однострочных тикетов, — это антипаттерн (каждый становится собственным конвейером + проверкой +
коммитами). Мелочь идёт в следующий коммит; связанные последующие задачи складываются в один пакет из 2–7
подзадач по общей поверхности; строка бэклога — это индекс, а не хранилище.

## 4. Режимы задач: процесс пропорционален задаче

Частая ошибка тяжеловесных конвейеров — утопить однострочное изменение в церемониях. tms-pipeline избегает
этого, классифицируя каждую задачу первым делом:

- **Direct** — косметика / текст / локальная правка, без изменения поведения во время выполнения →
  наименьшее связное изменение, дешёвая валидация. (Аудит пробелов может быть единственной строкой «skipped
  per minimal-surface exception».)
- **Investigation** — первопричина неясна → воспроизвести и проследить путь сбоя до исправления.
- **TDD-first** — поведение, логика, контракты, аутентификация, хранение, валидация → начать с падающего
  теста на уровне контракта/видимом пользователю, затем минимальная реализация до зелёного.

Работа над функциями по умолчанию идёт как TDD-first; тривиальная работа остаётся Direct. Полный механизм из
восьми стадий — для существенных задач, а не для исправления опечатки.

## 5. Как на самом деле выполняется многоагентная реализация (стадия 04)

Ведущая беседа **не** пишет код. Она пишет брифы волн, диспетчеризует агентов и обеспечивает соблюдение
ворот:

1. Классифицировать профиль сопровождения волны (A/B/C) и зафиксировать его.
2. Диспетчеризовать агента **Developer** с брифом волны (объём, файлы, приёмка).
3. Параллельно диспетчеризовать проверяющие роли для этого профиля: **Tester** всегда; **Architect** для
   B/C; **Security** для C; **Reviewer** всегда.
4. Волна проходит, только если каждая запущенная проверка зелёная (сборка/тесты/типы/линтер, нет дрейфа от
   дизайна, нет новых уязвимостей, соответствует плану + приёмке).
5. При сбое сфокусированный агент-исправитель устраняет конкретные находки; перезапускаются только упавшие
   ворота.
6. Переход к следующей волне — только после того, как все ворота пройдены.

Это держит контекстное окно каждого агента компактным (ведущий остаётся на высоком уровне, рабочие уходят
вглубь по одной волне) и ловит дрейф, регрессии и уязвимости *между* волнами, а не в самом конце. Коммиты не
несут **никакой ИИ-атрибуции** (лицензионное ограничение) и никогда не пушатся автоматически — ветка ждёт
проверки человеком и CI.

## 6. Почему это лучше одного мега-промпта

- Контекстное окно никогда не переполняется: каждая стадия и каждая волна получают только то, что им нужно.
- Ошибки ловятся там, где их исправить дешевле всего — неверный дизайн правится в тексте, а не в коде.
- Модели никогда не приходится гадать: к моменту реализации у неё есть утверждённый дизайн, конкретный план
  и стандарты вашего проекта.
- Самые тяжёлые усилия по проверке тратятся только там, где риск реален.
- Найденная работа и ручные шаги запуска фиксируются, а не теряются.

Методология — это живой инженерный процесс: адаптируйте названия стадий, триггеры сопровождения и промпты под
культуру вашей команды. Важен принцип: **контролируйте контекст на каждом шаге.**
