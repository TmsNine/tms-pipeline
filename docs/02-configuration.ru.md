# Конфигурация

> Как настроить tms-pipeline под ваш проект.
>
> tms-pipeline — это дисциплина для AI-агентов: она проводит одну уже поставленную задачу от тикета до
> проверенного кода, держа память агента чистой на каждом шаге. Чтобы делать это под *ваш* проект, агентам
> нужно знать ваши правила — где лежат задачи, на каком языке писать, какими командами проверять код. Всё
> это хранится в файлах настроек, о которых и рассказывает эта страница.
>
> Большую часть настройки за вас делает мастер-установщик (`npx tms-pipeline`) и скилл `/tms-init` — это
> команда, которую вы даёте агенту, чтобы он сам прочитал ваш репозиторий и заполнил настройки. Эта страница
> нужна для двух случаев: когда вы хотите настроить что-то вручную, и для значений, которые мастер
> намеренно оставляет пустыми (пометка `<<TODO>>` — заглушка, ждущая вашего решения). Английская версия —
> [02-configuration.md](02-configuration.md).

## Об инструментах: Claude Code и Codex

tms-pipeline работает поверх AI-инструмента — программы, в которой вы общаетесь с агентом. Поддерживаются
два: **Claude Code** и **Codex**. Вам нужен только один из них — выбирайте тот, которым уже пользуетесь.
Файл `AGENTS.md` общий для обоих, а Claude Code дополнительно читает свой `.claude/CLAUDE.md`. Различия
между инструментами собраны ниже, в разделе [«Паритет с Codex»](#codex).

## Где живут настройки: AGENTS.md и CLAUDE.md

Агенты не зашивают специфику вашего проекта в себя — они читают её на ходу из двух файлов настроек:

- **`AGENTS.md`** (в корне репозитория) — главный файл настроек проекта: где лежат задачи, на каком языке
  писать вывод, ваши правила. Это общий файл для обоих AI-инструментов (и Claude Code, и Codex).
- **`.claude/CLAUDE.md`** — правила, специфичные для одного из инструментов, Claude Code. В частности,
  он сопоставляет профили M/E/R/C с inline-работой, bounded evidence/test help или реальным proving-role
  mob. Этот файл подключает к себе `AGENTS.md` строкой `@./AGENTS.md`, поэтому всё, что вы написали
  в `AGENTS.md`, действует и здесь.

Начните с шаблонов `templates/AGENTS.template.md` и `templates/CLAUDE.template.md`. В них нужно заменить
каждый `{{PLACEHOLDER}}` на ваше значение и убрать поясняющие заметки `<!-- -->` / `« »` (мастер делает
это сам, если вы запускаете его).

## Значения, которые запрашивает мастер

Эти поля мастер выясняет короткими вопросами и заполняет за вас. В колонке справа — на что влияет каждое.

| Плейсхолдер | Значение |
|---|---|
| `OUTPUT_LANGUAGE` | Язык всего, что читает пользователь (отчёты, документы). |
| `AUDIENCE_PROFILE` | Кто читает вывод — от этого зависит уровень объяснений (например, нетехнический владелец vs старший инженер). |
| `PROJECT_ONE_LINER` | Что представляет собой проект и на каком стеке сделан. |
| `TASK_FOLDER_PATTERN` | Где лежат папки задач (например, `docs/<TICKET-ID>/`). |
| `DOC_BASE_PATH` | Путь к вашей базе документации (дерево docs, вики или Obsidian-vault). |
| `BACKLOG_LOCATION` | Файл бэклога — списка задач, который считается главным. |
| `TICKET_ID_FORMAT` | Формат идентификатора задачи, например `PROJ-123`. |
| `TEST_CMD` / `TYPECHECK_CMD` / `LINT_CMD` / `BUILD_CMD` | Ваши команды проверки кода: тесты, проверка типов, линтер, сборка. |
| `LAUNCH_PLAYBOOK_LOCATION` | Где вы ведёте список ручных действий перед запуском (playbook запуска). |

## Значения, которые вы заполняете вручную (мастер оставляет `<<TODO>>`)

Эти поля мастер не угадывает — они требуют вашего решения, и именно их важнее всего задать правильно.
Мастер оставляет на их месте пометку `<<TODO>>` (заглушку), чтобы вы её заметили и заполнили.

- **`PROFILE_C_TRIGGERS`** — список самых рискованных участков кода: тех, где легко сломать что-то важное и
  потому задаче нужен Profile R/C и более глубокое независимое ревью 04b. Имя placeholder оставлено ради
  совместимости со старыми шаблонами, но смысл теперь в риске и глубине проверки, а не в автоматическом
  запуске всех проверяющих во время реализации. Перечислите здесь те из перечисленного ниже, что есть у
  вас, и точные пути к модулям, где они лежат:
  - вход и авторизация — кто пользователь и что ему можно;
  - разграничение по арендаторам (арендатор — это отдельный клиент в общей системе; в однопользовательском
    приложении этого нет, тогда пункт пропускаете) и определение, кто именно сейчас в системе;
  - граница доверия — места, куда приходят данные снаружи и которые надо проверять;
  - секреты и подпись;
  - оплата;
  - пути, где ходят PII — личные данные пользователя (имя, почта, телефон).

  Именно этот список подгоняет проверки под *ваш* набор рисков: о перечисленных участках кода агенты будут
  знать, что там нужна усиленная проверка.
- **`PERSISTENCE_AND_TENANCY`** — как у вас хранятся данные и как система понимает, кто сейчас пользователь
  или арендатор, чтобы не смешать чужие данные.
- **`MIGRATION_POLICY`** — как вы вносите и фиксируете изменения схемы базы данных (если базы данных нет —
  удалите это поле).
- **`LAUNCH_STAGE_MAPPING`** — какой тип ручного действия в какой раздел вашего playbook запуска должен
  попадать.
- **`TRACEABILITY_LOCATION`**, **`CODE_LAYOUT_HINT`**, **`DOC_INDEX_HINT`**, **`DESIGN_SYSTEM_HINT`** —
  необязательные подсказки, которые помогают агенту ориентироваться в проекте (последняя — путь к
  дизайн-системе или библиотеке компонентов для UI-проектов). Удалите те, что к вам неприменимы.

> Не уверены, что писать? Не угадывайте в одиночку — попросите своего агента (Claude Code или Codex)
> прочитать ваш код и предложить значения, а вы их подтвердите или поправите.

## Серьёзность найденных проблем: классы A/B/C/D

На одной из стадий пайплайна (аудит пробелов) другой агент свежим недоверчивым взглядом ищет дыры в
готовом дизайне до написания кода. Каждую найденную дыру он относит к одному из четырёх классов по
тяжести: A (блокер), B (чинимая авария), C (мелкая доработка), D (теоретическая). Правила, что считать
каким классом, лежат в `AGENTS.md` с разумными значениями по умолчанию.

Подстройте примеры класса A под свой набор рисков. Например, приватный внутренний инструмент может убрать
формулировки про комплаенс, а у мультиарендного SaaS (где в одной системе данные многих клиентов) стоит
оставить классом A любую утечку данных между арендаторами.

## Паритет с Codex

Здесь — отличия Codex от Claude Code. Если вы пользуетесь только одним инструментом, читайте этот раздел
только при переходе на второй.

- Codex читает `AGENTS.md` напрямую — отдельно подключать его не нужно.
- Файл-манифест плагина (он сообщает инструменту, какие скиллы и команды есть) для Codex лежит в
  `.codex-plugin/plugin.json`, а для Claude Code — в `.claude-plugin/plugin.json`.
- Скиллы в обоих инструментах используют формат `SKILL.md`, но в этом репозитории два дерева: `skills/`
  для Claude Code и `codex-skills/` для Codex. Методология одна, но тексты не обязаны совпадать
  побайтно: у Codex пронумерованные имена стадий и Codex-native формулировки.
- Главное отличие workflow — стадия 04. Claude Code работает profile-aware: M остаётся inline, E использует
  bounded evidence/test help, а R/C получает реальное разделение proving-ролей. Codex по
  умолчанию делает mono/main-agent implementation с явными role self-checks. В обоих инструментах
  независимая стадия `04b_loop_review` отвечает за приёмку.
- Профиль волны в Codex — это сигнал риска и глубины ревью, а не просто список subagent-ов для запуска во
  время кодинга. Bounded-работа может остаться main-agent-only в 04; рискованная работа получает более
  глубокий 04b; максимальный риск всё ещё может идти через full classic multi-agent implementation, если
  оператор осознанно выбирает тяжёлый режим.
- Нативные роли лежат в разных деревьях: Claude Markdown-роли — в `agents/`, Codex TOML-роли — в
  `codex-agents/`. Намерение ролей `tms_*` остаётся общим, хотя форматы файлов и управление моделями
  у инструментов различаются. Claude agent-файлы задают `model` и `permissionMode`, а stage 04
  записывает фактический runtime evidence и не делает вид, что declarations точно применились.
  `permissionMode` работает у файлов, скопированных в project/user agent scope, но Claude Code игнорирует
  его у agents, загруженных прямо из plugin; plugin-run поэтому записывает parent/runtime permission evidence.

### Установка скиллов и агентов в Codex

У Codex нет команды вроде `/plugin install`, поэтому файлы нужно положить в `~/.codex` вручную или
полуавтоматически:

- **Полуавтоматически:** запустите `npx tms-pipeline`, ответьте «да» на вопрос «Do you use Codex?» и
  согласитесь на копирование — мастер скопирует `codex-skills/` → `~/.codex/skills/` и `codex-agents/` →
  `~/.codex/agents/`. Если вы не используете Codex, мастер каталог `~/.codex` не трогает.
- **Вручную:** `cp -R codex-skills/* ~/.codex/skills/ && cp -R codex-agents/* ~/.codex/agents/`.

Установщик сохраняет уже существующие шаблоны проекта, скиллы, команды и файлы ролей. Передавайте
`--force` только когда намеренно хотите заменить локальные настройки версиями из пакета.

Выбор моделей по стадиям — в [маршрутизаторе моделей](06-model-routing.ru.md).

Файл `AGENTS.md` при этом остаётся в корне проекта (его пишет мастер) — Codex читает его напрямую.
