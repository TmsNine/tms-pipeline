# Конфигурация

> Как настроить tms-pipeline под ваш проект. Мастер (`npx tms-pipeline` / `/tms-init`) заполняет большую
> часть этого за вас; эта страница — справочник для ручной настройки и для значений, которые мастер
> оставляет как `<<TODO>>`. Английская версия — [02-configuration.md](02-configuration.md).

## Канон: AGENTS.md + CLAUDE.md

Методология читает специфику вашего проекта во время выполнения из двух файлов:

- **`AGENTS.md`** (корень репозитория) — общий канон, читаемый Codex нативно и импортируемый Claude Code.
- **`.claude/CLAUDE.md`** — правила, специфичные для Claude Code (многоагентный «mob» реализации); он
  импортирует `AGENTS.md` через `@./AGENTS.md`.

Начните с `templates/AGENTS.template.md` и `templates/CLAUDE.template.md`. Замените каждый
`{{PLACEHOLDER}}` и удалите поясняющие заметки `<!-- -->` / `« »` (мастер делает это автоматически).

## Значения, которые запрашивает мастер

| Плейсхолдер | Значение |
|---|---|
| `OUTPUT_LANGUAGE` | Язык всего, что читает пользователь. |
| `AUDIENCE_PROFILE` | Кто читает вывод (например, нетехнический владелец vs старший инженер). |
| `PROJECT_ONE_LINER` | Что представляет собой проект + стек. |
| `TASK_FOLDER_PATTERN` | Где живут папки конвейера по задачам (например, `docs/<TICKET-ID>/`). |
| `DOC_BASE_PATH` | Путь к вашей базе документации. |
| `BACKLOG_LOCATION` | Файл бэклога (единый источник истины для задач). |
| `TICKET_ID_FORMAT` | Например, `PROJ-123`. |
| `TEST_CMD` / `TYPECHECK_CMD` / `LINT_CMD` / `BUILD_CMD` | Ваши команды валидации. |
| `LAUNCH_PLAYBOOK_LOCATION` | Где отслеживаются ручные предзапусковые действия. |

## Значения, которые вы заполняете вручную (мастер оставляет их как `<<TODO>>`)

Они требуют человеческого суждения — их важнее всего задать правильно:

- **`PROFILE_C_TRIGGERS`** — список поверхностей, которые принуждают к полному (Security) сопровождению во
  время реализации: ваша модель аутентификации/авторизации, разграничение по арендатору/разрешение
  идентичности, валидация ввода на границе доверия, секреты/подпись, платёжные поверхности, пути PII и
  точные пути модулей, в которых они расположены. Именно это делает модель стоимости соответствующей *вашему*
  профилю рисков.
- **`PERSISTENCE_AND_TENANCY`** — как хранятся данные и как разрешается и разграничивается идентичность
  арендатора/пользователя.
- **`MIGRATION_POLICY`** — как вносятся и регистрируются изменения схемы (удалите, если у вас нет базы
  данных).
- **`LAUNCH_STAGE_MAPPING`** — какой вид ручного действия идёт в какой раздел вашего playbook запуска.
- **`TRACEABILITY_LOCATION`**, **`CODE_LAYOUT_HINT`**, **`DOC_INDEX_HINT`**, **`DESIGN_SYSTEM_HINT`** —
  необязательные указатели (последний — путь к дизайн-системе/библиотеке компонентов для UI-проектов);
  удалите, если неприменимо.

Claude или Codex могут помочь вам составить их черновики, прочитав вашу кодовую базу, — просто попросите
после онбординга.

## Рубрика серьёзности

Рубрика серьёзности аудита пробелов (Класс A/B/C/D) поставляется с разумными значениями по умолчанию в
`AGENTS.md`. Подстройте примеры класса A под вашу модель рисков (например, приватный внутренний инструмент
может убрать формулировки про комплаенс; мультиарендный SaaS должен сохранить межарендную утечку данных как
класс A).

## Паритет с Codex

- Codex читает `AGENTS.md` нативно — отдельный импорт не нужен. Многоагентные правила `.claude/CLAUDE.md`
  применяются концептуально, но формулировки диспетчеризации через инструмент `Agent` специфичны для Claude
  Code.
- Манифест плагина для Codex лежит в `.codex-plugin/plugin.json`; для Claude Code это
  `.claude-plugin/plugin.json`.
- Навыки используют один и тот же формат `SKILL.md` в обоих инструментах. Тела навыков написаны независимо от
  инструмента (они описывают задачу, а не конкретный инструмент вроде Read/Write), потому что имена
  инструментов различаются между Claude Code и Codex.
- Кастомные субагенты: Claude Code читает определения агентов из `agents/` (и `~/.claude/agents/`); Codex
  использует собственный каталог агентов. Пять ролей «mob» в `agents/` переносимы по содержанию; разместите
  их там, где их ожидает каждый инструмент.

### Установка скиллов/агентов в Codex

У Codex нет аналога `/plugin install`, поэтому файлы нужно положить в `~/.codex`:

- **Автоматически:** запустите `npx tms-pipeline`, ответьте «да» на «Do you use Codex?» и согласитесь на
  копирование — мастер скопирует `skills/` → `~/.codex/skills/` и `agents/` → `~/.codex/agents/`. Если
  Codex вы не используете, мастер `~/.codex` не трогает.
- **Вручную:** `cp -R skills/* ~/.codex/skills/ && cp -R agents/* ~/.codex/agents/`.

`AGENTS.md` при этом остаётся в корне проекта (его пишет мастер) — Codex читает его нативно.
