# Ваша база документации

> Конвейер читает знания о продукте/архитектуре/бэклоге из «базы документации». Эта страница объясняет, как
> она устроена и как адаптировать включённые пустые скелеты. Английская версия —
> [03-doc-base.md](03-doc-base.md).

## Зачем нужна база документации

tms-pipeline — это методология *поставки*: ей нужно откуда-то читать, *что* представляет собой продукт и
*какую* задачу она поставляет. Этот источник истины — ваша база документации. Её путь задаётся как
`DOC_BASE_PATH` в `AGENTS.md`. Это может быть что угодно долговечное: дерево `docs/` в репозитории, вики,
хранилище Obsidian, экспорт из Notion. Методология никогда не предполагает конкретный инструмент — только
путь.

## Пустые скелеты

`templates/docs-vault/PROJECT_NAME/` — это готовая к заполнению структура, которую можно скопировать в вашу
базу документации:

```
PROJECT_NAME/            ← переименуйте эту папку в имя вашего проекта
├── README.md
├── 00 Governance/
│   ├── Definition of Ready and Done.md
│   └── Documentation System and Handoff Guide.md
├── 02 Product/
│   ├── PRD - FEATURE_AREA.md
│   └── Flow - FLOW_NAME.md
├── 03 Architecture/
│   └── Architecture Delta.md
└── 04 Delivery/
    ├── Backlog.md
    └── Traceability Map.md
```

Каждый файл — это обобщённый скелет с плейсхолдерами, без реального продуктового содержимого.

## Как их адаптировать

1. Скопируйте папку в вашу базу документации (мастер может сделать это, если вы ответите «да» на «copy
   docs-vault»).
2. **Переименуйте папку `PROJECT_NAME`** в имя вашего проекта.
3. Укажите на неё `DOC_BASE_PATH` в `AGENTS.md`.
4. Замените плейсхолдеры (`<FEATURE_AREA>`, `<FLOW_NAME>`, `<TICKET-ID>`) и наполните каждый скелет вашим
   реальным содержимым. Claude или Codex могут помочь — дайте им ваши существующие заметки и попросите
   заполнить скелеты PRD, флоу и бэклога.

## Для чего нужен каждый файл

- **Definition of Ready/Done** — входной/выходной порог для задачи. Конвейер проверяет «Ready» перед стадией
  00 и «Done» на контрольных воротах проверки.
- **Documentation System and Handoff Guide** — ваша иерархия источников истины и порядок чтения, чтобы новый
  участник (человек или агент) знал, где живёт авторитет.
- **PRD / Flow** — что делает продукт и как выполняются сценарии; стадии исследования и дизайна читают эти
  документы.
- **Architecture Delta** — архитектурная форма изменения; решения попадают сюда как заметки в стиле ADR.
- **Backlog** — единый источник истины для задач. Каждая строка — это **индекс, а не хранилище**: одна
  короткая строка, а более длинный контекст живёт в папке задачи. Пакеты полировки класса C из аудита
  пробелов регистрируются в разделе `### Bundled follow-ups`, никогда — в активной таблице.

## Держите легко

Строка бэклога — одна строка. Многоабзацные описания, пути файлов и состав живут в `00_ticket.md` папки
задачи. Не дробите бэклог на десятки тикетов — группируйте связанные последующие задачи (2–7 подзадач) по
общей поверхности. См. [методологию](00-methodology.ru.md).
