# Начало работы

> Подробный разбор. В [README](../README.ru.md) есть сжатый туториал; эта страница раскрывает требования,
> два способа подключения и установку. Английская версия — [01-getting-started.md](01-getting-started.md).

## Что такое tms-pipeline

tms-pipeline — это дисциплина для AI-агентов: она проводит одну уже поставленную задачу от тикета до
проверенного кода, держа память агента чистой на каждом шаге. Вы устанавливаете её поверх существующего
проекта, после чего даёте агенту команды (скиллы) — а он по шагам доводит задачу до готового кода.

Несколько слов, которые встретятся дальше:

- **скилл** — это команда вида `/tms-research`, которую вы даёте агенту, чтобы он сделал один шаг работы.
- **контекстное окно** — рабочая память агента: всё, что он сейчас видит и держит в голове. Чем больше в
  ней лишнего, тем хуже ответ. Главная идея пайплайна в том, чтобы на каждом шаге давать агенту только
  нужное, а не вываливать всю задачу разом.
- **AGENTS.md** — файл с настройками вашего проекта: где лежат задачи, на каком языке писать, команды
  тестов, ваши правила. Агенты читают его, чтобы знать специфику именно вашего репозитория.

Важно сразу понять, чем tms-pipeline **является** и чем **не является**:

- **Является** процессом, который ведёт одну задачу через поэтапные проверки до кода, готового к
  слиянию.
- **Не является** генератором проектов: он не придумывает функции, не строит приложение с нуля и не
  заменяет ваши продуктовые решения. Что строить — решаете вы.

Работа в основном идёт «на бумаге»: все стадии, кроме 04, дают текстовый документ (`.md`) — план,
дизайн, цикл ревью, отчёт. Код появляется только на одной стадии — 04 (реализация). Полный обзор стадий —
[в разборе стадий](04-stages-deep-dive.ru.md); там же видно, какие агенты и модели работают на каждой.

## Требования — прочитайте сначала

Поскольку tms-pipeline ставится **поверх существующего проекта**, прежде чем начать, у вас должны быть:

- репозиторий с настоящим исходным кодом;
- база документации — место, где постоянно живут знания о продукте и архитектуре: дерево `docs`, вики или
  Obsidian-vault, где угодно;
- в идеале — бэклог (список задач, которые предстоит сделать).

Если у вашего проекта пока нет документации или бэклога, в комплекте есть пустые заготовки
(`templates/docs-vault/`) — они дают стартовую структуру, но наполнять их реальными продуктовыми
решениями придётся вам, и делаете вы это так, как принято у вас.

### Если проекта ещё нет

Описанное выше нужно, когда проект уже существует. Если же вы стартуете совсем с нуля, есть отдельный
скилл **`/tms-new`** — разовая первичная подготовка нового продукта. Он проводит вас через интервью —
по одному вопросу за раз — и в конце раскладывает ваши ответы в стартовый набор документов и бэклог.

`/tms-new` не придумывает продукт за вас: что строить, решаете вы, а он лишь раскладывает ваши ответы по
местам. Это разовый запуск, а не одна из стадий пайплайна. Подробности — в [README](../README.ru.md).

> **Какой скилл выбрать.** Проекта ещё нет (ни кода, ни задач) → `/tms-new`. Проект уже есть (код и/или
> документация на месте) → `/tms-init` (см. ниже). Если запустить `/tms-init` на пустом репозитории, он
> сам предложит начать с `/tms-new`.

## Два способа подключения

Оба бесплатны и используют одно и то же ядро, не зависящее от инструмента.

### A. «Под ключ» — установщик, затем ваш агент

Подключение намеренно разбито на две половины. Сначала работает **установщик** — обычная программа в
терминале, которая просто кладёт файлы на место. Затем подключается **агент**: скилл `/tms-init`
дочитывает ваш репозиторий и заполняет настройки.

```bash
# 1. Установщик — выбрать язык + инструмент(ы), поставить скиллы, создать стартовый AGENTS.md.
#    Он НЕ спрашивает про проект (команды тестов, формат тикета, пути к докам).
npx tms-pipeline
#    На шаге установки есть два способа разложить скиллы. Используйте ОДИН из них, иначе скиллы
#    задвоятся:
#      - либо установщик копирует скиллы в ~/.claude / ~/.codex (он предложит сам);
#      - либо, только для Claude Code, через маркетплейс плагинов (команды ниже).
#    Берите маркетплейс, если хотите получать обновления через /plugin; копирование — если хотите
#    держать файлы у себя и обновлять вручную.
/plugin marketplace add TmsNine/tms-pipeline
/plugin install tms-pipeline@tms-pipeline
/reload-plugins
```

```text
# 2. Внутри Claude Code / Codex — доводим настройку чтением репозитория:
/tms-init
```

`/tms-init` сканирует репозиторий (команды тестов и сборки из `package.json` / настроек CI, паттерны
Ticket-ID из истории git, существующие доки и бэклог) и **заполняет за вас `AGENTS.md`** — тот самый файл
настроек проекта. Спрашивает он только то, что не смог вычитать сам, — в первую очередь где находится ваша
база документации.

Несколько полей требуют вашего решения, и угадать их по коду нельзя (например: на каких участках кода
нужна усиленная проверка, есть ли в системе изоляция данных по клиентам, как у вас принято менять схему
базы). Для них `/tms-init` отправляет в гайд [ручной настройки](05-manual-setup.ru.md) с готовыми
промптами.

### B. «Вручную» — подключить всё самостоятельно

Если вы предпочитаете сначала понять методологию и подключить всё руками:

1. Прочитайте [методологию](00-methodology.ru.md).
2. Установите плагин (две команды `/plugin` выше) или скопируйте папки `skills/` и `agents/` в ваш
   `~/.claude/` (и `~/.codex/skills/` для Codex).
3. Скопируйте `templates/AGENTS.template.md` в ваш репозиторий как `AGENTS.md` и заполните плейсхолдеры
   вручную (см. [конфигурацию](02-configuration.ru.md)).

> Не уверены, что писать в плейсхолдеры? Не угадывайте в одиночку — попросите своего AI-агента (Claude
> Code или Codex) прочитать ваш код и предложить значения, а вы их подтвердите или поправьте.

## Проверка установки

В Claude Code выполните `/help` или откройте список скиллов — вы должны увидеть скиллы `tms-*`. Затем
запустите задачу (см. туториал в README).

## Codex

Claude Code и Codex — это два AI-инструмента, в которых может работать пайплайн; вам нужен только один из
них. Codex читает `AGENTS.md` напрямую, так что методология работает точно так же, а сами скиллы от
инструмента не зависят. Аналога `/plugin install` у Codex нет, поэтому скиллы и агентов нужно положить
туда, где Codex их ищет, — в `~/.codex`:

- проще всего: запустите `npx tms-pipeline`, выберите Codex и согласитесь на копирование — установщик сам
  скопирует `codex-skills/` → `~/.codex/skills/` и `agents/` → `~/.codex/agents/`;
- вручную: `cp -R codex-skills/* ~/.codex/skills/ && cp -R agents/* ~/.codex/agents/`.

Различия в каталогах см. в [конфигурации](02-configuration.ru.md#codex).

## Посмотреть пример

В [`templates/example-task/ACME-101/`](../templates/example-task/ACME-101/) лежит полный синтетический
прогон одной задачи через поэтапный пайплайн, включая цикл ревью 04b:

- `00_ticket.md` — тикет (постановка задачи);
- `01_research.md` — исследование;
- `02_design.md` — дизайн;
- `02b_gap_audit.md` — аудит пробелов;
- `03_delivery_plan.md` — план реализации;
- `04_implementation.md` — реализация (единственная стадия, где появляется код);
- `04b_loop_review.md` — независимый цикл ревью/фикса;
- `05_test_report.md` — отчёт о тестировании;
- `06_review_gate.md` — финальная проверка.

Прочитайте файлы по порядку, чтобы увидеть формат каждой стадии до того, как запускать пайплайн на своей
задаче.

→ Что именно делает каждая стадия под капотом (какие агенты, на каких моделях, где именно ваша проверка) —
в [docs/04-stages-deep-dive.ru.md](04-stages-deep-dive.ru.md).
