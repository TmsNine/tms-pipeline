# Stage 06 — Review gate

Read THIS project's `AGENTS.md` (Codex reads it natively) for specifics: task-folder path, launch-playbook location, output language.

> **Model tier.** Prefer Terra high for straightforward closure and Sol high/xhigh for Profile R/C, partial validation, unresolved manual gates or a possible no-go. Fallbacks: Terra → `gpt-5.4`, Sol → `gpt-5.5`. Luna may summarize known evidence but never issues the final verdict. Never use Fast mode. If the runtime cannot prove the selected model, record actual model as runtime-selected/unknown.

For fingerprints, read and run the canonical helper from the sibling `../tms-04-implement/references/task-fingerprint.mjs`. Require helper version `tms-task-fingerprint-v1`; never recreate the hash with an ad-hoc command.

## Method

1. **Start from artifacts, not a fresh wide review.** Read `02_design.md`, `03_delivery_plan.md`, `04_implementation.md`, `04b_loop_review.md`, and `05_test_report.md`. Treat accepted `04b_loop_review.md` as the independent code-review evidence; do not redo a full 04b-style review from scratch unless the 04b artifact is missing, skipped, failed, or contradicted by later evidence.
2. **Verify implementation against the design contract** (`02_design.md`) and acceptance criteria (`03_delivery_plan.md`). This is implementation-vs-design and launch-readiness judgement — distinct from `02b_gap_audit` (design-vs-risks) and `04b_loop_review` (independent code review/fix loop).
3. **Confirm:** acceptance criteria met; no design drift; the normalized status field in 04b is exactly `PASS` (`NOT_ACCEPTED`, `SKIPPED`, `NEEDS_REMEDIATION` and `BLOCKED` cannot receive go/conditional_go, regardless of optimistic prose elsewhere); change-surface triggers handled; R/X ledger resolved; V-ID evidence present; docs aligned; follow-ups consolidated; pre-launch manual actions captured. Derive the complete changed-path set from Git state (`git diff --name-only -z --no-renames <base_sha> --` plus `git ls-files --others --exclude-standard -z`), classify every path as task-owned or unrelated, and fail closed on ambiguity/overlap. `--no-renames` is mandatory so a rename remains source deletion + destination addition before and after staging. Materialize the observed task-owned set and require exact equality with the package manifest through the helper's `--observed` input. Then recalculate implementation/package fingerprints in `worktree` mode. Require the implementation hash to match accepted 04b and fresh stage-05 evidence. A task is NOT done if the symptom is gone but the same mechanic stays structurally inconsistent across coupled layers.
4. **Verdict:** `go` / `conditional_go` (name the manual condition and ensure it is recorded in the launch playbook, not just here) / `no-go` with reasons and the best next experiment.
5. Write `docs/<TASK-ID>/06_review_gate.md` with checks, evidence pointers, 04b status, validation status, launch/manual conditions, verdict, and closing-commit eligibility. Do not put a future commit SHA in this repo-local artifact: a commit cannot contain its own final SHA.
6. **Close the task in the external delivery docs (mandatory before `go` / `conditional_go` is complete).** The review gate is not finished when `06_review_gate.md` is written. Update and reread the owning external backlog/status and launch-playbook entries. External documentation-base files are outside the Git package and must be reported as separately verified. If mandatory external sync cannot be completed and verified, stage 06 is `BLOCKED`; do not issue a successful verdict or commit.

   **The backlog row is an index, not a storage location — keep it to ONE short line (≤200 chars).** Set status to `Done` / `Done (conditional_go)` (or leave open on `no-go`) with: the date, the verdict, a ≤1-line plain-language phrase of *what shipped* (actually delivered scope, not the original ticket wording if it was narrowed), and a `См. docs/<TASK-ID>/` pointer. **Do NOT paste the review summary, sub-item lists, migration numbers, condition lists, file paths, or follow-up chains into the backlog row.** That full detail already lives in `06_review_gate.md` (verdict + conditions) and in the launch playbook (manual steps). On `no-go`, leave the task open and note in one line what blocks closure.
7. **Create exactly one closing task commit on `go` / `conditional_go`.** The pipeline invocation authorizes this one commit. It contains all and only repo-local task-owned changes from stages 00–06. It does not contain external documentation-base files and does not imply that `conditional_go` launch actions have already happened. `no-go`, stage `BLOCKED`, failed external sync, mismatched fingerprints, ambiguous task ownership, overlapping dirty paths, or unrelated staged files forbid the commit.

   Compare the canonical package fingerprint and complete observed path set before staging. Stage only the manifest paths that clearly belong to `<TASK-ID>`. Derive the staged path set with `git diff --cached --name-only -z --no-renames <base_sha> --`; require it to equal the package manifest exactly, pass that set through `--observed`, then run the same helper with `source=index` and require its package hash to equal the pre-stage `source=worktree` hash. Never stage unrelated user changes, generated noise, neighboring task edits, or a pre-existing user index. If a task-owned file mixes unrelated hunks, do not guess: leave the commit unmade and report the exact blocker. Use a concise task-prefixed message, for example `<TASK-ID>: close pipeline task`. Never push automatically and never add `Co-Authored-By:` or any AI/agent attribution. After the commit succeeds, report the actual SHA in chat or another external status surface; do not edit `06_review_gate.md` afterward merely to insert that SHA.

8. Add a compact `Pipeline metrics` block (maximum 12 rows) to `06_review_gate.md`: execution profile; waves completed/planned; review/fix rounds and first→final score; findings A/B/C/D found/fixed/deferred; remediation cycles; fresh/reused/failed V-IDs plus unique/repeated validation commands; implementation fingerprint match; primary-signal status; open manual gates; efficiency (`stage time | subagents by role | owner interventions | words across nine artifacts | input/output/cache tokens`, using `n/a` when unavailable); verdict. Metrics never override an unresolved Class A/B finding.

The final summary must name which external doc was updated and verified, the verdict, whether the one closing commit was created, and if not, which exact condition prevented it.
